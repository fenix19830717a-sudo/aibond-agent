# aibond - 企业人机协同路由平台 设计文档

> 日期：2026-06-12
> 状态：已批准

## 1. 项目概述

aibond 是一个企业级人机协同聊天平台，核心理念是让 AI Agent 作为平等的聊天参与者加入企业协作。用户可以通过 Web UI 集中管理和对话分布在不同设备、不同架构上的 AI Agent，同时支持群组协作和工作流驱动的业务流程。

### 核心定位

- **不是企业AI服务**：aibond 是消息路由和协作平台，专注于转发消息给多个不同的 Agent
- **Agent 是主动参与者**：Agent 可主动发起对话、加入群组、注册 Skills，而非被动等待触发
- **人机协同节点**：每个工作节点 = 人审核 + AI LT（Lightweight Task）

### 核心功能

1. 用户登录平台后对话自己的 AI Agent（信息转发 + 穿透）
2. AI Agent 通过唯一密钥（MCP 和 Skills）自动注册和登录平台，更新状态和地址
3. 群组模式：设置聊天群组，加入成员（人 + Agent）
4. 工作流模式：可视化拖拽画布定义业务流程，节点 = 人审核 + AI LT
5. 内容共享：文件/知识库共享、消息搜索、Agent 状态监控

## 2. 架构设计

### 2.1 整体架构（中心化消息 Broker）

```

│                      Web UI (React)                      │

│  └──────────┘ └──────────┘ └──────────┘ └────────────┘ │
└──────────────────────┬──────────────────────────────────┘
                       │ WebSocket + REST API
┌──────────────────────▼──────────────────────────────────┐
│                  aibond Server (FastAPI)                  │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────────┐ │
│  │ 认证模块  │ │ 消息路由  │ │ 工作流引擎│ │ Agent注册  │ │
│  └──────────┘ └──────────┘ └──────────┘ └────────────┘ │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐                │
│  │ 群组管理  │ │ 心跳管理  │ │ 内容共享  │                │

                       │

              ▼        ▼        ▼
         ┌────────┐┌────────┐┌────────┐
         │PostgreSQL││ Redis  ││ 对象存储│
         │(持久化) ││(状态/  ││(文件/  │
         │         ││ 消息队列││ 知识库) │
         └────────┘└────────┘└────────┘
```

### 2.2 核心设计原则

1. **Server 无状态化**：业务逻辑无状态，状态全部存 Redis，支持水平扩展
2. **消息驱动**：模块间通过 Redis 消息队列通信，解耦合
3. **自适应心跳**：根据 Agent 在线活跃度动态调整心跳间隔（5s~60s）
4. **MCP/Skills 作为 Agent 接入协议**

### 2.3 技术选型

| 层级 | 技术 | 理由 |
|------|------|------|
| 前端 | React + TypeScript + Ant Design | 成熟 UI 库，组件丰富 |
| 前端画布 | React Flow | 工作流可视化拖拽的事实标准 |
| 后端 | Python FastAPI | 异步高性能，AI 生态友好 |
| WebSocket | FastAPI WebSocket | 实时通信 |
| 数据库 | SQLite（MVP）→ PostgreSQL（生产） | MVP 阶段零依赖 |
| 缓存/队列 | Redis | 状态存储、消息队列、发布订阅 |
| 认证 | JWT | 渐进式，先 JWT 后扩展 OAuth |

## 3. Agent 注册与通信

### 3.1 Agent 接入方式

支持两种方式：
- **主动注册**：Agent 通过唯一密钥主动连接 aibond Server，类似 MCP 协议注册
- **手动添加**：用户在平台上手动添加 Agent，配置 API 地址和密钥

### 3.2 通信模式（混合模式）

平台做注册发现 + 初始连接建立，后续消息根据情况选择中转或直连。

**三层网络保障**：

1. **WebSocket 长连接（主通道）**：Agent 与 Server 维持 WS 长连接，消息实时送达
2. **主动地址更新（辅助通道）**：Agent 网络变化时主动更新地址，Server 记录最新可达地址
3. **消息缓存 + 离线队列**：Agent 离线期间消息存 Redis，重连后自动推送

### 3.3 网络变化处理

```
Agent IP 变化
├─ WebSocket 仍在线 → 无需处理，消息继续通过 WS 送达
├─ WebSocket 断开 → Agent 检测网络变化
│   ├─ 立即用新 IP 重连 Server
│   ├─ 重连成功 → 拉取离线消息 → 恢复正常
│   └─ 重连失败 → 指数退避重试（1s, 2s, 4s...最大 60s）
└─ Server 端：WS 断开 → 标记可能离线 → 尝试回调 URL 推送
    ├─ 推送成功 → Agent 通过 HTTP 接收消息
    └─ 推送失败 → 消息入离线队列，等 Agent 重连
```

### 3.4 自适应心跳

| Agent 活跃度 | 心跳间隔 | 超时判定 | 状态 |
|-------------|---------|---------|------|
| 高频交互     | 5 秒    | 15 秒   | 在线 |
| 正常使用     | 15 秒   | 45 秒   | 在线 |
| 低频使用     | 30 秒   | 90 秒   | 空闲 |
| 长时间无交互  | 60 秒   | 180 秒  | 离线 |

### 3.5 Agent 注册协议

```json
{
    "agent_id": "agent-uuid",
    "api_key": "unique-secret-key",
    "name": "代码审查助手",
    "skills": ["code_review", "git_operations"],
    "mcp_endpoints": ["mcp://localhost:3001"],
    "callback_url": "https://agent.example.com/webhook",
    "capabilities": {
        "accepts_websocket": true,
        "accepts_webhook": true,
        "accepts_polling": false
    }
}
```

通信优先级：WebSocket 长连接 > HTTP 回调 > 轮询拉取

## 4. 群组聊天与消息路由

### 4.1 群组模型

```
群组 (Group)
├── 基本信息：ID、名称、描述、创建者
├── 成员列表
│   ├── 人 (User)：普通成员、管理员
│   └── AI Agent：带 Skills 标签、在线状态
├── 消息流
│   ├── 文本消息、文件/图片消息
│   ├── 系统消息（成员加入/离开）
│   └── 工作流触发消息
└── 设置：消息可见性、Agent 权限、通知规则
```

### 4.2 消息路由流程

```
发送者 → Server 接收 → 验证 → 持久化 → 路由决策 → 投递
                                          ├── DM → 路由到目标
                                          ├── 群组 → fan-out
                                          └── @提及 → 路由到被@的 Agent
```

### 4.3 Agent 在群组中的行为

- **被动响应**：被 @提及时处理消息并回复
- **主动参与**：可主动在群组中发送消息
- **权限控制**：管理员可设置 Agent 是否可主动发言、是否仅响应 @提及

## 5. 工作流引擎

### 5.1 节点类型

| 节点类型 | 说明 |
|---------|------|
| 触发节点 | 手动触发 / 群组消息触发 / 定时触发 |
| AI 执行节点 | 指定 Agent 执行任务 |
| 人工审核节点 | 人审核并决定通过/拒绝/修改 |
| 条件分支节点 | 根据上一步结果选择路径 |
| 输出节点 | 发送到群组 / 通知 / 记录 |

### 5.2 可视化画布

使用 React Flow 构建拖拽画布，节点面板提供 5 种节点类型，连线定义执行顺序。

### 5.3 工作流执行

1. 触发（手动/消息/定时）
2. 执行 AI 节点 → Server 发送任务给 Agent，等待结果
3. 执行人工审核节点 → 工作流暂停，群组中发送审核请求
4. 条件分支 → 根据输出自动选择路径
5. 输出 → 结果发送到关联群组

### 5.4 工作流与群组交互

- 工作流可在群组中创建和触发
- 审核节点在群组中以对话形式呈现
- 输出自动发送到关联群组
- 群组消息可触发工作流

## 6. 内容共享与管理

- **文件共享**：上传文件，群组内共享
- **消息搜索**：全文检索消息历史
- **Agent 状态监控**：仪表盘展示在线状态、消息量、响应延迟

## 7. 性能优化策略

| 策略 | 实现方式 | 预期效果 |
|------|---------|---------|
| 自适应心跳 | 活跃 5s，空闲 60s | 减少 80%+ 无效心跳 |
| 消息压缩 | WebSocket gzip | 减少 60%+ 带宽 |
| 增量同步 | 客户端记录 last_msg_id | 减少重复传输 |
| 消息 TTL | 离线消息 7 天过期 | 控制存储增长 |
| 懒加载 | 消息历史按需加载 | 减少数据库查询 |
| Server 无状态 | 水平扩展 + 负载均衡 | 线性扩展能力 |
